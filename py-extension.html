The Python extension of Maude SE provides metaSmtCheck and metaSmtSearch functions. We use Maude-as-a-library to implement our extension.

This is an alpha version.

Installation

You can install our Python extension from pre-built Python wheels (Linux, MacOS).

Download and unzip the zip file and go to the unzipped directory.

~$ unzip maude-se-py-linux.zip && cd maude-se-py-linux

The directory includes two wheels (one for the extended Maude-as-a-library and the other for our extension interface). Use the following command to install the both wheels.

~/maude-se-py$ pip install *.whl

We built and tested our extension on Ubuntu 18.04.

Use the following command to run and test our examples.

~/maude-se-py$ python3 run_examples.py

Features

metaSmtCheck

We can check the satisfiability of SMT queries with metaSmtCheck (similar to the metaCheck) by importing the META-SMT-CHECK module in the smt-check.maude file. The usage of metaSmtCheck is similar to the metaCheck function except that it takes a boolean flag as its last argument, where the flag decides whether to generate a satisfying assignment for the satisfiable case.

--- META-SMT-CHECK module ...
op metaSmtCheck : Module Term -> SmtCheckResult .
eq metaSmtCheck(M:Module, TERM:Term) = metaSmtCheck(M:Module, TERM:Term, false) .

op metaSmtCheck : Module Term Bool -> SmtCheckResult
  [special (id-hook SpecialHubSymbol                   (SmtCheckerSymbol)
            op-hook unknownResultSymbol                (unknown : ~> SmtCheckResult)
            ...
            term-hook builtinTrueTerm                  ((true).Bool)
            term-hook builtinFalseTerm                 ((false).Bool)
    )] .

The check result is given as a term of sort SmtCheckResult, which can be one of unknown, true, false, or {_}. The symbol {_} consists of a set of assignments, where each assignment is denoted as a term _|->_.

sort SmtCheckResult .
subsort Bool < SmtCheckResult .

op unknown : -> SmtCheckResult [ctor] .
op {_} : SatAssignmentSet -> SmtCheckResult [ctor] .

sort SatAssignment .
op _|->_ : Term Term -> SatAssignment [ctor] .

sort SatAssignmentSet .
subsort SatAssignment < SatAssignmentSet .
op empty : -> SatAssignmentSet [ctor] .
op _,_ : SatAssignmentSet SatAssignmentSet -> SatAssignmentSet [ctor comm assoc id: empty] .

The following shows a simple example usage of metaSmtCheck.

load smt
load smt-check

fmod SIMPLE is
  pr INTEGER .
  pr META-SMT-CHECK .
endfm

red metaSmtCheck(upModule('INTEGER, false), upTerm(X:Integer > 4), true) .
==========================================
reduce in SIMPLE : metaSmtCheck(upModule('INTEGER, false), upTerm(X:Integer > (4).Integer), true) .
rewrites: 1
result SmtCheckResult: {'X:Integer |-> '5.Integer}

Currently, we do not support metaSimplify. We will update this later.

Uninterpreted symbols

The metaSmtCheck can handle SMT formulas with uninterpreted symbols. We use Maude operators to declare uninterpreted symbols and annotate them as uninterpreted symbols using metadata attributes. For example, we can declare uninterpreted symbol f having the domain and range sort A as follows.

sort A .
op f : A -> A [ctor metadata "smt euf"] .

The metadata smt euf tells our extension that the operator is an SMT symbol (smt) and is an uninterpreted symbol (euf). The following is an example module containing the uninterpreted symbol f and an equality.

fmod EUF is
  pr BOOLEAN .

  sort A .

  op _===_ : A A -> Boolean .
  op f : A -> A [ctor metadata "smt euf"] .
endfm

fmod EUF-CHECK is
  pr EUF .
  pr META-SMT-CHECK .

  vars X Y : A .
endfm

We can check the satisfiability of the formula containing the symbol f with smtCheck as usual.

red smtCheck(f(f(X)) === X and f(X) === Y and not X === Y) .
==========================================
reduce in EUF : smtCheck(f(f(X)) === X and f(X) === Y and not X === Y) .
z3 model -->
[X = A!val!1,
 Y = A!val!0,
 f = [A!val!1 -> A!val!0, else -> A!val!1]]
rewrites: 2
result Bool: (true).Bool

Currently, we do not support generating an assignment for the formula with uninterpreted function. Instead, we print the result model.

The following shows another example.

fmod EUF-XOR is
  pr INTEGER .
  op _xor_ : Integer Integer -> Integer [metadata "smt euf"] .
endfm

fmod EUF-XOR-CHECK is
  pr EUF-XOR .
  pr META-SMT-CHECK .
  var N : Integer .
endfm

red metaSmtCheck(upModule('EUF-XOR, false), upTerm(not(N === 0) and 0 <= N and 0 <= (N xor 255) 
                                                    and N < 256 and not (0 <= (N xor 255)))) .
red metaSmtCheck(upModule('EUF-XOR, false), upTerm(0 <= (N xor 255))) .

Interpreted symbols

The metaSmtCheck can also check the satisfiability of SMT formulas with interpreted symbols from various theories. We use Maude operators to declare interpreted symbols and annotate them as interpreted symbols using metadata. For example, we can declare interpreted symbol select of the array theory with domain and range sort Integer as follows.

sort Array{Integer,Integer} .
op select : Array{Integer,Integer} Integer -> Integer [metadata "smt array:select"] .

The metadata smt array:select tells our extension that the operator is an SMT symbol (smt) and connected to the select symbol of the array theory. We can connect the interpreted symbol select of the Array theory to two different operators (select and _[_]).

op _[_] : Array{Integer,Integer} Integer -> Integer [metadata "smt array:select"] .

The following module contains all the symbols with an equality.

fmod EUF-ARRAY is
  pr REAL-INTEGER .

  sort Array{Integer,Integer} .

  op _===_ : Array{Integer,Integer} Array{Integer,Integer} -> Boolean .
  op select : Array{Integer,Integer} Integer -> Integer [metadata "smt array:select"] .
  op _[_] : Array{Integer,Integer} Integer -> Integer [metadata "smt array:select"] .
  op store : Array{Integer,Integer} Integer Integer -> Array{Integer,Integer} [metadata "smt array:store"] . 
endfm

fmod EUF-ARRAY-CHECK is
  pr EUF-ARRAY .
  pr META-SMT-CHECK .

  var A1 : Array{Integer,Integer} .
  vars X Y : Integer .
endfm

We can check the satisfiability of formulas under EUF and Array theories as follows.

red metaSmtCheck(upModule('EUF-ARRAY, false), upTerm(select(A1, X) === X and store(A1, X, Y) === A1)) .
red metaSmtCheck(upModule('EUF-ARRAY, false), upTerm(A1[X] === X and store(A1, X, Y) === A1)) .

==========================================
reduce in EUF-ARRAY-CHECK : metaSmtCheck(upModule('EUF-ARRAY, false), 
                                         upTerm(select(A1, X) === X and store(A1, X, Y) === A1)) .
z3 model -->
[A1 = Store(K(Int, 2), 3, 3), Y = 3, X = 3]
rewrites: 4 in 4ms cpu (3ms real) (1000 rewrites/second)
result Bool: (true).Bool
==========================================
reduce in EUF-ARRAY-CHECK : metaSmtCheck(upModule('EUF-ARRAY, false), 
                                         upTerm(A1[X] === X and store(A1, X, Y) === A1)) .
z3 model -->
[A1 = Store(K(Int, 2), 3, 3), Y = 3, X = 3]
rewrites: 4 in 4ms cpu (3ms real) (1000 rewrites/second)
result Bool: (true).Bool

As expected, the satisfiability results of both formulas are the same.

In the later version, we support parametric modules and theory modules for defining Array-like parametric sorts.

metaSmtSearch2

We provide the metaSmtSearch2 function to perform search using rewriting modulo SMT. The function is similar to the metaSmtSearch function except that its last boolean flag enables folding.

op metaSmtSearch2 : Module Term Term Condition Qid Bound Nat Bool -> SmtResult? .
eq metaSmtSearch2(M, T, T', COND, STEP, B, N, FOLD)
= metaSmtSearchInternal(transform(M), makeConstTerm(T), makeConstTerm(T'), COND, STEP, B, N, FOLD) .

op metaSmtSearch2 : Module Term Term Condition Qid Bound Nat -> SmtResult? .
eq metaSmtSearch2(M, T, T', COND, STEP, B, N)
= metaSmtSearch2(M, T, T', COND, STEP, B, N, false) .

op metaSmtSearchInternal : Module Term Term Condition Qid Bound Nat Bool -> SmtResult?
  [special (id-hook SpecialHubSymbol      (SmtSearchSymbol)
            op-hook failureSymbol         (failure : ~> SmtResult?)
            op-hook resultSymbol          ({_,_,_,_} : Term Substitution Term Nat ~> SmtResult))].

We can use the function by importing the META-SMT-SEARCH2 module in the smt-search.maude file. The following shows a GCD example for rewriting modulo SMT.

load smt
load smt-search

mod GCD is
  pr INTEGER .

  sorts GcdResult .
  op gcd : Integer Integer -> GcdResult [ctor] .
  op return : Integer -> GcdResult [ctor] .

  vars X Y : Integer .

  crl [r1] : gcd(X, Y) => gcd(X - Y, Y) if X > Y = true [nonexec] .
  crl [r2] : gcd(X, Y) => gcd(X, Y - X) if X < Y = true [nonexec] .
  crl [r3] : gcd(X, Y) => return (X) if X === Y  = true [nonexec] .
endm

Currently, we assume that the condition of each rule does not contain any non-SMT constraint. We will allow non-SMT conditions later.

mod GCD-ANALYSIS is
  pr GCD .
  pr BOOL .
  pr META-SMT-SEARCH2 .

  vars I J : Integer .
endm

The following shows a robot example.

omod ROBOT is
  pr REAL .

  class Robot | pos : Vector, vel : Vector, acc : Vector, time : Real .

  sort Vector .
  op [_,_] : Real Real -> Vector [ctor] .
endom

omod ROBOT-DYNAMICS is
  pr ROBOT .

  vars O : Oid .
  vars CONST CONST' : Boolean .
  vars PX VX AX PY VY AY PX' VX' AX' PY' VY' AY' T T' TAU : Real .

  crl [move]:
      < O : Robot | pos  : [PX,  PY], vel  : [VX,  VY],
                    acc  : [AX,  AY], time : T >
   => < O : Robot | pos  : [PX', PY'], vel  : [VX', VY'],
                    time : T + TAU >
   if TAU >= 0/1 and VX' === VX + AX * TAU and VY' === VY + AY * TAU
  and PX' === 1/2 * AX * TAU * TAU + VX * TAU + PX
  and PY' === 1/2 * AY * TAU * TAU + VY * TAU + PY = true [nonexec] .

  crl [accX]:
      < O : Robot | acc : [AX,  AY] >
   => < O : Robot | acc : [AX', AY] >
   if AY === 0/1 = true [nonexec] .

  crl [accY]:
      < O : Robot | acc : [AX, AY] >
   => < O : Robot | acc : [AX, AY'] >
   if AX === 0/1 = true [nonexec] .

  op r : -> Oid [ctor] .
endom

mod ROBOT-ANALYSIS is
  pr ROBOT-DYNAMICS .
  pr BOOL .
  pr META-SMT-SEARCH2 .

  vars IPX IPY IVX IVY IAX IAY CLK NPX NPY : Real .
  var ATTRSET : AttributeSet .
endm

Python API

As explained, we use Maude-as-a-library to implement our extension. We use the same Python implementation and expose metaSmtCheck to a Python function (smtSearch2). For example, we can perform the same analysis in Python as follows.

import maude
...
from rwsmt.api import *

maude.init(advise=False)
maude.load(os.path.join(os.path.dirname(__file__), '.', 'api-test.maude'))
gcd = maude.getModule('GCD')

init = gcd.parseTerm('gcd(10, I:Integer)')
goal = gcd.parseTerm('return(J:Integer)')

c = maude.EqualityCondition(gcd.parseTerm('I:Integer < 9 and I:Integer > 0'), 
							gcd.parseTerm('(true).Bool'))

gcd_result = smtSearch2(gcd, init, goal, c, "=>*", "unbounded", 7, False)
print(gcd_result)

The arguments of the smtSearch2 function are not meta-level terms.
